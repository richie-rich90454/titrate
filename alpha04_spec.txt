3. LLVM BACKEND ARCHITECTURE
================================================================================

3.1 Integration Point
- A new Rust module `trc/src/codegen/llvm/` will be created.
- The compiler driver selects the backend via a command‑line flag
  (`--native` or `--release`).  Default remains the bytecode VM.

3.2 Type Mapping (Titrate → LLVM)
| Titrate Type | LLVM Type |
|--------------|-----------|
| void         | void      |
| bool         | i1        |
| byte         | i8        |
| short        | i16       |
| int          | i32       |
| long         | i64       |
| vast         | i128      |
| uvast        | u128      |
| float        | f32       |
| double       | f64       |
| half         | half      |
| quad         | fp128     |
| char         | i32       |
| string       | { i64, i8* }   (length + UTF‑8 pointer) |
| size         | i64       |
| Owned<T>     | T*        (heap pointer; drop flag managed) |
| &T / &mut T  | T*        (bare pointer) |
| array<T>     | { i64, T* }   (length + elements pointer) |
| Class        | { vtable*, field0, field1, … } |
| Interface    | { object*, vtable* }   (fat pointer) |
| Enum         | { i32 tag, [max‑variant‑bytes] } |
| Tuple        | anonymous LLVM struct |

3.3 Ownership & Memory Management in LLVM IR
- **Owned<T> drop flag**: For each local Owned<T>, an `i1` alloca is
  created.  On move, the source drop flag is set to false.  At scope
  exit, if the drop flag is true, the destructor (if any) and `free()`
  are called.
- **Borrows**: `&T` / `&mut T` are raw pointers with no drop logic.
- **Regions**: Lowered to `alloca` + `lifetime.start`/`lifetime.end`.
  Escape analysis already prevents pointers from leaving the region.
- **`unsafe`**: Emits raw IR; `malloc`/`free` from libc.

3.4 Virtual Dispatch
- Vtable globals for each class; virtual calls load function pointer
  from vtable via GEP.  Interfaces use fat pointers.

3.5 String Interpolation
- Already desugared by the parser into `Binary(Add)` chains.  The LLVM
  backend uses a runtime function `titrate_string_concat` (provided by
  the native support library).

================================================================================
4. BRIDGING THE @native CALLS – EXACT MECHANISM
================================================================================

Since **all** native functions now use real Rust crates and have no stubs,
we can create a single static library `libtitrate_native.a` that wraps
the existing `trc/src/bytecode/vm/natives/` functions with C ABI.

4.1 Creating the Native Support Library
   - Add a new crate `titrate_native` (or modify the VM’s `natives`
     module) to expose `#[no_mangle] pub extern "C"` functions for every
     native name used by the standard library (e.g. `titrate_File_readLine`,
     `titrate_Math_sin`, …).
   - The wrapper converts between C‑compatible types (pointers, integers)
     and the internal `Value` types.  For most numeric/string operations,
     the conversion is simple (e.g., a Titrate string is a length+pointer
     struct, which can be passed as two C parameters).
   - The crate is compiled to a static library and installed alongside
     the compiler.

4.2 LLVM Backend Calls
   - The LLVM backend maintains a mapping table from @native names to
     the corresponding C function signatures.  When an `@native("X")`
     is encountered, the backend emits a direct call to `titrate_X`.
   - The linker (invoked by `pipette build --native`) links against
     `libtitrate_native.a` and all system libraries (libc, OpenSSL, etc.).

4.3 Guaranteed Behavioural Equivalence
   - The exact same Rust functions that run in the bytecode VM are
     invoked from native code, ensuring identical results for all
     standard library calls.

================================================================================
5. COMPILATION PIPELINE
================================================================================

The `trc` binary gains a `--native` flag.  When invoked:
1. Lex, parse, analyze the source (identical to bytecode path).
2. Run the LLVM backend to produce `.o` files.
3. Call the system linker to produce the final executable, linking with:
   - `libtitrate_native.a`
   - The C runtime (`libc`, `libm`, etc.)
   - Optional system libraries (e.g. `-lssl`, `-lcrypto`)

`pipette build --native` automates the entire process, reading linker
flags from `Titrate.toml`.
